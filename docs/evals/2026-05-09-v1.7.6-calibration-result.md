## Calibration-script bug discovered + fixed in v1.7.6

The starvation guard (post-review P1-4) was structurally inert. `runOneBudget` read `j.conditions[condName].results[]` to count per-task `memoryRecalled` flags, but `run.mjs::buildOutput` (line 393) does not serialize the per-task `results` array in single-seed JSON — it only writes `overall_trap_hit_rate`, `phases`, `learns`, `improvement_pct`, and `hook_failures`. The extraction returned `[]`, the guard false-positive flagged every candidate as `starved=true`.

**The bug did not affect the verdict.** `lateMean=0%` across all 5 budgets is the load-bearing rejection signal — no candidate would satisfy the band even with a working starvation guard. The bug was masked by the structural floor.

**Fix shipped in v1.7.6:** `runOneBudget` no longer attempts the broken extraction. Returned row shape: `{budget, seedRates, lateMean, lateCI}`. `selectBStar` still respects an optional `starved` field defensively, but rows no longer carry it. `calibration-result.json` was re-derived from the existing per-seed JSONs after the fix; phase data was unchanged.

**Tracked for v1.7.7+:** "Expose per-task results in single-seed JSON" + "re-enable starvation guard" with the corrected schema.
