### Task A3: Tenant-isolation test for scorerWindow (testing INFO #5)

**Files:**
- Create: `tests/api-recall-scorer-window-tenant-isolation.test.ts`

**Step 1: Write the test file**

```typescript
/**
 * v1.7.1 INFO #5 — scorerWindow does not leak across tenants.
 *
 * The store-side LIMIT inside loadSearchEntries is applied AFTER the tenant
 * predicate (`WHERE tenant_id = ?`). A future refactor that moved the LIMIT
 * before the tenant filter would silently surface cross-tenant rows when
 * scorerWindow > tenant row count. Pin the contract.
 */

import { describe, it, expect, beforeEach, afterEach } from 'vitest';
import { mkdtempSync, mkdirSync, rmSync } from 'node:fs';
import { tmpdir } from 'node:os';
import { join } from 'node:path';
import { initStore, writeEntry } from '../src/store.js';
import { createMemory, Layer, MemoryKind, type MemoryEntry } from '../src/memory.js';
import { recall, type Context } from '../src/api.js';

function makeRoot(prefix: string): string {
  const root = mkdtempSync(join(tmpdir(), `hippo-${prefix}-`));
  mkdirSync(join(root, '.hippo'), { recursive: true });
  initStore(root);
  return root;
}
function safeRmSync(p: string): void {
  try { rmSync(p, { recursive: true, force: true }); } catch { /* best-effort */ }
}
function ctxFor(root: string, tenantId: string): Context {
  return { hippoRoot: root, tenantId, actor: 'test:f3-tenant' };
}
function makeRaw(text: string, tenantId: string): MemoryEntry {
  return createMemory(text, {
    layer: Layer.Buffer,
    confidence: 'observed',
    kind: 'raw' as MemoryKind,
    tenantId,
  });
}

describe('scorerWindow tenant isolation (v1.7.1 INFO #5)', () => {
  let root: string;
  beforeEach(() => { root = makeRoot('f3-tenant'); });
  afterEach(() => safeRmSync(root));

  it('wide scorerWindow under one tenant does not surface rows from another', () => {
    const N = 5;
    // Tenant A: N rows matching 'kappa'.
    for (let i = 0; i < N; i++) writeEntry(root, makeRaw(`kappa A-${i}`, 'tenant-a'));
    // Tenant B: N rows ALSO matching 'kappa'. If the LIMIT moved before
    // the tenant predicate, scorerWindow=N+5 under tenant-a could pull
    // tenant-b rows.
    for (let i = 0; i < N; i++) writeEntry(root, makeRaw(`kappa B-${i}`, 'tenant-b'));

    const result = recall(ctxFor(root, 'tenant-a'), {
      query: 'kappa',
      limit: 100,
      scorerWindow: N + 5,
    });
    // Every returned row must come from tenant-a (no B rows).
    for (const r of result.results) {
      expect(r.content).not.toContain('B-');
      expect(r.content).toContain('A-');
    }
    // Total candidates capped at the tenant population, NOT scorerWindow.
    expect(result.total).toBe(N);
  });
});
```

### Task A5: Replace fragile LIKE-fallback test with `DROP TABLE memories_fts` + content anchor (senior P2.3 + INFO #7)

**Background.** The existing `LIKE fallback path: substring query that misses FTS but matches LIKE...` test (tests/store-bm25-score.test.ts:97-114) relies on FTS5's word-boundary tokenizer to skip a substring match. Senior P2.3 from the v1.7.0 review flagged this as tokenizer-fragile: a future migration to a different tokenizer (e.g. `porter` or `trigram`) would silently skip the LIKE branch and the test would still "pass" by hitting FTS instead. INFO #7 separately flagged that the assertion does not anchor on content, so a partial FTS hit could let it pass spuriously.

**Both fix the same failure mode. Combine into one task.**

**Files:**
- Modify: `tests/store-bm25-score.test.ts` — replace the existing `LIKE fallback path: substring query that misses FTS...` test.

**Step 1: Replace the test**

```typescript
  it('LIKE fallback path: explicitly drop memories_fts → LIKE branch returns expected row, bm25_score undefined (senior P2.3 + INFO #7)', () => {
    // Senior P2.3: the prior test depended on FTS5's default unicode61
    // tokenizer skipping the substring "alphab" against word "alphabet".
    // Tokenizer-dependent; would silently skip the LIKE branch under
    // porter/trigram tokenizers. Robust route: drop memories_fts entirely
    // so loadSearchRows falls through to the LIKE branch deterministically.
    //
    // Plus INFO #7: anchor the assertion on the expected content so a
    // partial FTS hit cannot let the test pass spuriously.
    writeEntry(root, makeRaw('alphabet soup contents'));

    // Drop the FTS table out from under the store. loadSearchRows checks
    // FTS availability via setMeta('fts5_available'), but the actual
    // SELECT against memories_fts will throw if the table is gone — so
    // the implementation must catch and fall through. Verify behaviour.
    const { DatabaseSync } = require('node:sqlite') as typeof import('node:sqlite');
    const db = new DatabaseSync(join(root, '.hippo', 'memory.db'));
    try {
      db.exec('DROP TABLE IF EXISTS memories_fts');
      // Mark fts5_available=0 so loadSearchRows takes the LIKE branch by
      // intent (defensive: the meta-flag check runs before any SELECT).
      db.exec("INSERT OR REPLACE INTO meta (key, value) VALUES ('fts5_available', '0')");
    } finally {
      db.close();
    }

    const results = loadSearchEntries(root, 'alphab', 200, 'default');
    expect(results.length).toBeGreaterThan(0);
    // Anchor: the expected row must come back through the LIKE branch.
    expect(results.some((e) => e.content === 'alphabet soup contents')).toBe(true);
    // No FTS path → no bm25_score.
    for (const e of results) {
      expect(e.bm25_score).toBeUndefined();
    }
  });
```
