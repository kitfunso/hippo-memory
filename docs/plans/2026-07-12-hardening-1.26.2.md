## Problems

1. **Value-less `--scope` misbehaves inconsistently across 7 commands (14 sites in cli.ts).**
   parseArgs stores a value-less flag as boolean `true`; the consumer sites then split into
   three idioms:
   - `String(flags['scope']).trim()` → the literal string `'true'` becomes a real scope input:
     cmdRemember:771 (the `scope:true` TAG dual-write), cmdRecall:917 (the recall filter/unlock
     input), cmdExplain:1985, cmdContext:5481, cmdWm:6671 (`?? 'default'` does not save it —
     `true` is not nullish) /6695/6724/6733 (the four wm subcommand branches: push/read/clear/
     flush — the flag is read inside each branch, not once per command).
   - Silent drop (`typeof === 'string'` guards that coerce to null/undefined): cmdRemember:744 —
     the envelope WRITE, the worst variant (user believes the memory was scope-protected; it was
     written with scope NULL, and in the same command the tag half writes `scope:true`) — plus
     cmdAssemble:6822 and **the `case 'remember':` THIN-CLIENT dispatch at cli.ts:8216**
     (corrected from round 1's "cmdSlack" misattribution: the site sits in the top-level switch
     inside the `runViaServerIfAvailable` relay at 8212, reachable only when `detectServer`
     finds a live `hippo serve`; when no server is up the case falls through to `cmdRemember`).
   - Exactly one correct guard: cmdImport:6018 errors on non-string/empty.
   Owning commands (7): remember (744 + 771 direct, 8216 thin-client), recall, explain,
   context, import, wm (4 subcommand branches), assemble. `--hops` and `--max-neighbors`
   already treat value-less as a usage error; `--scope`, the security-relevant flag, does not.
2. **tests/server-concurrency.test.ts ECONNRESET flake under full-suite parallel load** (sole
   suite failure in 3 consecutive episodes; 2/2 full runs failed 2026-07-11; passes in
   isolation). Hypothesis (UNMEASURED — capture is a mandatory execute task): starved-event-loop
   keep-alive reuse race — ~100 in-flight kept-alive fetches against one Node server whose
   `keepAliveTimeout` is the 5s default while the suite saturates the host; the server closes an
   idle socket exactly as undici reuses it. Structurally consistent: serve() (server.ts:3227)
   sets no `keepAliveTimeout`/`headersTimeout`; the test's own comment history already conceded
   this class once (read-chunking added for SYN-burst ECONNREFUSED). Production exposure is soft:
   the thin-client silently falls back to direct mode on fetch failure (H2, server.ts:293-303),
   so server hardening is standard posture, not an urgent prod bug.

## Tasks

### T1 — ONE GLOBAL dispatch-level `--scope` guard (round-2 redesign)

Instead of applying a guard at 14 consumer sites, validate the flag ONCE, immediately after
argv parsing in the CLI entry point, BEFORE the top-level command switch:

```ts
/** Global --scope well-formedness guard (v1.26.2). parseArgs stores a value-less
 *  flag as boolean true; downstream the 14 consumer sites either coerced that to
 *  the literal scope string 'true' (recall filter/unlock input, wm session scope,
 *  the remember scope-tag dual-write) or silently dropped the user's scoping
 *  intent (the remember envelope WRITE). Reject it once here, mirroring the
 *  --hops value-less guard, so every current and future command - including the
 *  thin-client dispatch relays - sees --scope only as a non-empty string. */
if ('scope' in flags && (typeof flags['scope'] !== 'string' || !flags['scope'].trim())) {
  console.error('--scope requires a non-empty value (e.g. --scope slack:private:C1).');
  process.exit(1);
}
```

Placement: in the main dispatch function, after `parseArgs`, before the `switch` — the executor
locates the exact spot and confirms EVERY `flags['scope']` consumer is downstream of it
(including case-'remember' 8216 and any other `runViaServerIfAvailable` relay that forwards
scope). The 14 consumer sites are then left UNTOUCHED (surgical): their `typeof === 'string'`
narrowing becomes provably sound, the silent-drop branches become unreachable for the boolean
case, and cmdImport's bespoke guard stays as redundant-but-harmless defense (its error message
may be pinned by tests; do not touch it). This resolves the round-1 cmdWm question (no
per-branch application at all) and the 8216 reachability question (the guard fires pre-dispatch,
so exit-1 tests need NO live server).

Enumeration check: executor repo-greps `flags\['scope'\]` across the repo to confirm no consumer
exists OUTSIDE the guarded dispatch path (counting note: cli.ts:6030 contains the token twice on
one line — count logical sites, not occurrences; the known set is 14 tokens / 13 logical sites /
7 commands). Known parallel path, confirmed NON-issue (plan-critic round 2): src/mcp/server.ts
reads `args.scope` from JSON-RPC tool args at 459/831/977 — a separate transport that cannot
produce parseArgs's boolean coercion and already validates `typeof === 'string' && length > 0`;
no change there.

Deliberate consequence (CHANGELOG-called-out): a value-less or empty/whitespace `--scope` now
exits 1 on EVERY hippo command, including commands that ignore the flag today - well-formedness
is validated globally, matching how a mistyped flag should behave. No change for valued
`--scope`. Error string: plain text, no em dashes (frontend-string rule).
